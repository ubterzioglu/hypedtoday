# HYPED.today Test Plan: Auth, Admin, Add Project, Edge Functions ve Deploy Dogrulamasi

## Mevcut Durum

**Var olan test altyapisi:**
- Vitest + jsdom + @testing-library/react (vitest.config.ts)
- 5 test dosyasi: auth-redirect, combo-detection, dto-sanitization, request-limits, security-regression
- Test setup: `src/test/setup.ts` (sadece `@testing-library/jest-dom`)
- Playwright yok, E2E katmani yok

**Kapsanan dosyalar:**
- `src/lib/auth.tsx` – AuthProvider, useAuth hook, session/profile yonetimi
- `src/lib/api.ts` – getAuthHeaders, apiCall wrapper, tum API metodlari
- `src/lib/auth-redirect.ts` – normalizeBaseUrl, normalizePath, getAuthRedirectUrl
- `src/components/ProtectedRoute.tsx` – loading, profileResolved, requireAdmin gating
- `src/pages/AdminLogin.tsx` – next parametreli login, auto-redirect
- `src/pages/AuthCallback.tsx` – code exchange + next redirect
- `src/App.tsx` – routing (ProtectedRoute ile /add-project, /admin, /my-reviews, /my-claims)
- Edge functions: request-limits, create-post, admin-dashboard, admin-actions (+ _shared auth, cors, rate-limit, response, supabase, validation)
- Migrations: profiles admin RLS fix (is_admin SECURITY DEFINER helper)

### Adim 1 – Vitest Unit/Component Testlerini Genislet

**Hedefler:** Auth redirect, ProtectedRoute, AdminLogin, api.ts mock akislari

**Dosya: `src/tests/auth-redirect.test.ts` (genisletme)**
- `getAuthRedirectUrl` icin yeni testler:

  - `nextPath` undefined/empty/invalid ise "/" doner

- Mevcut normalizeBaseUrl/normalizePath testleri korunur

**Dosya: `src/tests/protected-route.test.ts` (yeni)**
- ProtectedRoute component'i test edilecek:
  1. `loading=true, session yok` -> spinner gosterir
  2. `loading=false, user null` -> `/admin/login?next=...` yonlendirmesi
  3. `loading=false, user var, requireAdmin=false` -> children render
  4. `requireAdmin=true, session var, profileResolved=false` -> spinner (admin bekleme)
  5. `requireAdmin=true, user.role='admin'` -> children render
  6. `requireAdmin=true, user.role='user'` -> `/` yonlendirmesi
- `useAuth` mock'lanacak, react-router-dom MemoryRouter ile sarilacak

**Dosya: `src/tests/admin-login.test.ts` (yeni)**

  4. `user` varsa ama `profileResolved=false` ise bekler (render null)
  5. Magic link formu: bos email -> toast error, gecerli email -> signInWithMagicLink cagrilir

**Dosya: `src/tests/api.test.ts` (yeni)**
- `getAuthHeaders` mock testleri:

**Dosya: `src/tests/auth-callback.test.ts` (yeni)**

**Yaklasim:** Edge function'larin handler mantigini test etmek icin shared modulleri vitest ile test edecegiz. Function index.ts dosyalari `Deno.serve` kullandigi icin dogrudan vitest'te calismaz, bunun yerine:

**Dosya: `supabase/functions/_shared/__tests__/cors.test.ts` (yeni)**
- `corsHeaders` testleri:
  1. Izin verilen origin -> Access-Control-Allow-Origin set
  2. Bilinmeyen origin -> bos string
  3. Extra headers merge edilir
- `handleCors` testleri:
  1. OPTIONS + izinli origin -> 204 + CORS headers
  2. OPTIONS + bilinmeyen origin -> 403
  3. GET -> null (CORS response donmez)
- `originCheck` testleri:
  1. Origin yok -> true
  2. Izinli origin -> true
  3. Bilinmeyen origin -> false

**Dosya: `supabase/functions/_shared/__tests__/validation.test.ts` (yeni)**
- validateRequired, validateMaxLength, validateUrl, validateEnum, validateUuid, collectErrors
- Her fonksiyon icin positive + negative case

**Dosya: `supabase/functions/_shared/__tests__/response.test.ts` (yeni)**

- errorResponse: dogru status, error sarmalama, code opsiyonel

**Not:** Auth ve rate-limit modülleri Supabase client bagimliligi oldugu icin mock-based integration test olarak Adim 3'te ele alinacak.

### Adim 5 – E2E Test Katmani (Playwright)

**Kurulum:**
```bash
npm install -D @playwright/test
npx playwright install chromium
```

**Dosya: `playwright.config.ts` (yeni)**
- baseURL: staging URL (env degiskeni)
- Test dizini: `e2e/`
- Timeout: 30s
- Retry: 1 (flaky test icin)
- Browser: chromium only

**Dosya: `e2e/auth-redirect.spec.ts` (yeni)**
1. Anonymous -> `/add-project` acar -> login sayfasina redirect
2. Login sayfasinda next parametresi korunur
3. Callback simuelasyonu: cookie/token ile `/auth/callback?next=/add-project`
4. Sonrasinda `/add-project` sayfasinda user gorunur

**Dosya: `e2e/add-project.spec.ts` (yeni)**
1. Authenticated user -> `/add-project` acilir
2. RequestCapacityWidget yuklenir (401 vermez)
3. Form alanlari gorunur ve doldurulabilir
4. En az bir task secili
5. Gecerli LinkedIn URL ile submit
6. Success toast gorunur
7. `/showroom` sayfasina yonlenir

**Dosya: `e2e/admin.spec.ts` (yeni)**
1. Admin user -> `/admin` acilir, dashboard yuklenir
2. Flags tab -> liste veya bos-state gorunur
3. Settings tab -> input degerleri okunabilir
4. Bir settings degeri degistirilip kaydedilir
5. Audit log'da action gorunur

**Dosya: `e2e/permission-boundary.spec.ts` (yeni)**
1. Normal user -> `/admin` acar -> erisemez (redirect veya denial)
2. Admin user -> `/add-project` acar -> erisebilir

**Dosya: `e2e/helpers/auth.ts` (yeni)**

- Admin/normal user fixture'lari

## Onemli Notlar

1. **Mock-based yaklasim:** Tum testler mock'larla calisacak. Staging/prod Supabase'e baglanti gerektiren testler sadece smoke check (Adim 6) ve RLS manual checklist (Adim 4) ile yapilacak.
2. **Google login otomasyonu:** Tam otomasyon zor. Callback simuelasyonu test yardimcilari ile yapilacak; gercek Google akisi manual smoke olarak korunacak.
3. **Edge function Deno serve:** Dogrudan vitest'te calistirilamaz. Handler mantigi shared modul testleriyle, cagri zinciri ise mock-based integration testleriyle kapsanacak.
4. **Mevcut testler:** Mevcut 5 test dosyasi oldugu gibi korunacak, sadece auth-redirect.test.ts genisletilecek.
5. **Test kullanicilari:** Staging'de mevcut admin + normal user fixture'lari Playwright E2E (Adim 5) tarafindan kullanilacak.
6. **Yeni dosya sayisi:** ~14 yeni test dosyasi + 3 config/script dosyasi
